# 流程

## 1. 准备

- 从 `main` 分支创建功能分支。
- 明确本次变更目标与影响范围。
- 确认目标合并分支（`main` 或 `dev`）与当前迭代一致。

## 2. 开发

- 按最小改动原则实现需求。
- 保持提交信息清晰，按功能拆分提交。

## 3. 自检

- 本地检查变更内容与影响文件。
- 确认没有引入无关改动。
- 必要时补充文档与配置示例。

## 4. 提交 PR

- 发起 Pull Request 到本次迭代指定分支（通常为 `main` 或 `dev`）。
- 使用 `.github/pull_request_template.md` 完整填写：
  - 背景 / 变更摘要
  - 测试命令与结果
  - 配置与部署变更
  - 风险与回滚
- 确保 PR 标题简明，推荐格式：
  - `cloud: ...`
  - `gateway: ...`
  - `docs: ...`

## 5. 评审与合并

- 根据评审意见迭代修改。
- CI 通过后合并。

## PR 文案建议

- 一句话说明“为什么做这次改动”。
- 变更摘要优先写“行为变化”，少写实现细节。
- 测试部分优先贴可复现命令。
- 若涉及配置文件或部署脚本，明确“升级步骤”和“回滚步骤”。
