# 部署说明

## 架构概览

AI 推理服务以 Docker 容器方式部署，通过 HTTP/JSON 接收 Rust 云端后端的请求。

```
Rust Cloud Backend ──HTTP──▶ AI Engine Container (FastAPI + PyTorch)
                              ├─ POST /api/v1/predict
                              └─ GET  /api/v1/health
```

## 训练环境

- 使用 Kaggle 或本地 Python 环境完成训练
- 安装开发依赖：`pip install -r requirements-dev.txt`

## 推理服务部署

### 方式一：Docker Compose（云端推荐，强烈建议采用）

> [!IMPORTANT]
> **绝对不需要在宿主机安装依赖**：部署测试只需要有 Docker 即可。无需在云服务器宿主机使用 `conda` 或 `pip` 安装庞大的 PyTorch 和其他环境依赖，Docker 容器已经完全隔离并打包好了所需的一切。

```bash
# 1. 确保训练好的模型存在于指定目录
# 必须包含 best_model.pth，config.yaml 以及 labels.json
# cp best_model.pth models/rice_leaf_classifier/best_model.pth

# 2. 构建镜像 (基于代码及环境打包)
docker compose build

# 3. 启动服务（-d 表示后台持续运行）
# 容器配置了自动重启，即使服务器重启也会自动恢复服务
docker compose up -d

# 4. 冒烟测试：验证服务健康状态
# 推荐纯终端测试：直接在云端终端运行命令判断是否畅通
curl http://localhost:8000/api/v1/health

# 5. E2E 推理测试：使用绝对路径或自动查找图片测试
# 注意：curl 命令是在宿主机执行，通过 8000 端口将图片以网络流传给 Docker 内部
TEST_IMAGE=$(find local_data -name "*.jpg" | head -n 1)
curl -X POST http://localhost:8000/api/v1/predict -F "file=@$TEST_IMAGE"

# 6. 监控运行日志（排障神器）
docker compose logs -f ai-engine

# 7. 停止容器
docker compose down
```

### 部署与测试避坑指南 (必读)

> [!WARNING]
> **网页无法加载 / Swagger Docs 加载白屏？**
> 如果你在使用 VS Code Remote SSH，VS Code 尝试将云端的 8000 端口转发到本地电脑浏览器时，极易与你本地已有的开发环境产生端口冲突，导致 `http://127.0.0.1:8000/docs#/` 无法正常加载。
> 
> **解决方案**：放弃网页测试，关闭 VS Code 底部面板的端口转发功能，直接在云端终端内使用 **纯终端调用（如上方第4、5步的 `curl` 命令）** 进行验证和开发联调！这种方式最敏捷且零干扰。

### 方式二：本地 uvicorn 直接运行

```bash
# 安装依赖
pip install -r requirements-dev.txt

# 启动
uvicorn service.main:app --reload --host 0.0.0.0 --port 8000
```

## 环境变量

| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| `MODEL_CHECKPOINT_PATH` | 模型权重文件路径 | `models/rice_leaf_classifier/best_model.pth` |
| `MODEL_LABELS_FILE` | 标签文件路径 | `models/rice_leaf_classifier/labels.json` |
| `MODEL_CONFIG_FILE` | 模型配置路径 | `models/rice_leaf_classifier/config.yaml` |

## Docker 镜像设计

- **多阶段构建**：builder 阶段安装依赖，runtime 阶段仅复制产物，镜像更精简
- **CPU-only PyTorch**：推理不需要 CUDA，镜像约 ~1.2GB（vs CUDA 版 ~6GB）
- **非 root 用户**：容器内以 `appuser` 运行，安全最佳实践
- **模型 Volume 挂载**：权重文件通过 volume 挂载，换模型无需重建镜像
- **健康检查**：Docker HEALTHCHECK 调用 `/api/v1/health`

## 依赖管理

| 文件 | 用途 |
|------|------|
| `requirements.txt` | 运行时依赖（进 Docker） |
| `requirements-dev.txt` | 开发依赖（含 torch、pytest 等） |

> 注意：`torch` 和 `torchvision` 不在 `requirements.txt` 中，由 Dockerfile 通过 CPU-only 索引单独安装。
